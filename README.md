# ASUTP / AKB5

## Что это

`AKB5` это Windows-десктопное приложение на C# / .NET 8 WinForms для ведения древовидной базы знаний по АСУТП.

Проект хранит структуру по цехам, отделениям, оборудованию и связанным объектам. Основной внутренний формат хранения на текущем этапе это JSON.

- `Program.cs` — точка входа

- `Forms/InputDialog.cs` — отдельный диалог ввода строки

- `Models/SavedData.cs` — корневая модель сохранения со `SchemaVersion`

- `Services/KnowledgeBaseService.cs` — базовые операции над деревом знаний

## Что было найдено в исходной версии

Ключевые проблемы исходной выгрузки до правок:

1. Был риск потери данных, потому что корректное полное сохранение не выполнялось при обычном закрытии.
2. `paste` и `drag-and-drop` некорректно работали с поддеревьями и глубиной уровней.
3. Reload выполнялся без безопасного dirty-check.
4. Undo-история ограничивалась неправильно и теряла не те снапшоты.
5. Инициализация пустой базы была хрупкой.
6. Почти вся логика была сосредоточена в `MainForm.cs`.
7. Поиск мог устаревать после мутаций дерева.
8. Техническая гигиена проекта была слабой: артефакты сборки, отсутствие тестов, отсутствие нормальной документации и build-контура.

## Решения по развитию

### JSON или Excel

Excel не стоит делать основным хранилищем для этой задачи.

Рациональная схема на текущем этапе такая:

- source of truth: JSON
- user-facing exchange format: Excel

Значит Excel разумно добавлять как импорт/экспорт, но не как primary storage.

### Когда думать о SQLite

Переход с JSON на SQLite имеет смысл, если появятся:

- большие объёмы данных
- несколько операторов
- аудит изменений
- более сложный поиск и фильтрация
- более строгие требования к устойчивости хранения

Пока JSON ещё допустим, но следующий серьёзный шаг зрелости это скорее SQLite, чем Excel как основной формат.

Следующий практический шаг теперь такой:

1. Вынести из формы orchestration `undo/redo` и post-restore синхронизации дерева.
2. После этого дочистить presentation-часть `load`-результатов и transient UI reset.
3. Сразу после каждого вынесенного блока добавлять тесты на новую не-UI логику.
4. После закрытия этого хвоста `MainForm` сделать полный Windows UI-smoke как ближайшую контрольную точку.
5. Только после этого расширять функциональность вроде Excel-импорта/экспорта.

## Когда будет полный тест

Под полным тестом в этом проекте сейчас понимается не только `dotnet build` и `dotnet test`, а реальный Windows-smoke:

- открыть существующую базу
- переключить цех
- добавить, удалить и переименовать узлы
- проверить `copy/paste`
- проверить `drag-and-drop`
- проверить `save`, `save as`, `reload`
- проверить `undo/redo`
- закрыть и заново открыть приложение

Этот полный тест запланирован как ближайшая контрольная точка сразу после завершения текущего хвоста декомпозиции `MainForm`.
То есть порядок теперь такой:

1. закрыть оставшиеся выносы по `undo/redo` и `load`-orchestration
2. поднять чистую Windows `11 ARM` VM
3. выполнить полный smoke и зафиксировать результат в этом файле

Иными словами, полный тест будет не "когда-нибудь потом", а сразу после ещё `1-2` инженерных выносов из `MainForm`, до Excel и до новых пользовательских фич.

## Конкретный план действий

Ниже практический порядок работы без распараллеливания лишних задач.

### Шаг 1. Закрыть публикацию и CI

Цель:

- убедиться, что репозиторий `arbbuz/AKB5` стал рабочей базой проекта, а не просто архивом файлов

Действия:

- открыть GitHub Actions и проверить первый запуск `windows-build.yml`
- если CI красный, разобрать причину и исправить её до любых новых фич
- после зелёного CI считать инженерную базу подтверждённой

Статус на сейчас:

- первый запуск `Windows Build` в GitHub Actions уже завершился успешно

Результат шага:

- есть удалённый репозиторий с подтверждённой сборкой и тестами

### Шаг 2. Продолжить декомпозицию на macOS

Цель:

- не держать архитектурную работу заблокированной из-за проблемной Windows VM

Действия:

- убрать остатки прикладной orchestration-логики из `MainForm`
- дочистить `undo/redo` и post-restore sync поверх уже вынесенных сервисов
- вынести presentation-логику `load`-результатов из прикладного хвоста формы настолько, насколько это возможно без ломки UI
- после каждого выноса добавлять тесты на новый не-UI слой

Результат шага:

- `MainForm` продолжает уменьшаться
- дальнейшая разработка перестаёт зависеть от текущего состояния VM

### Шаг 3. Подтвердить реальную работоспособность UI на Windows

Цель:

- проверить фактический пользовательский сценарий уже после следующего раунда декомпозиции

Порядок:

- поднять или пересоздать корректную чистую Windows VM
- установить `.NET 8 SDK`
- собрать и запустить `AKB5`
- проверить минимум: открытие базы, переключение цехов, add/delete/rename, copy/paste, drag-and-drop, save, reload, undo/redo

Результат шага:

- либо подтверждённый рабочий UI-smoke
- либо список конкретных дефектов, которые мешают использовать приложение

### Шаг 4. Расширить тестовый контур после декомпозиции

Цель:

- закрепить вынесенную логику тестами до новых пользовательских фич

Действия:

- покрыть новые service/controller-компоненты
- добавить edge-case тесты для backup/recovery
- подготовить базу для будущих migration-тестов по `SchemaVersion`

Результат шага:

- новые изменения защищены тестами, а регрессии ловятся раньше

### Шаг 5. Улучшить файловый UX и восстановление данных

Цель:

- убрать самые неприятные сценарии для пользователя до расширения функциональности

Действия:

- сделать понятнее аварийную загрузку из backup
- при необходимости улучшить поведение первого запуска и выбора файла
- продумать сценарий частично валидных данных без резкого отказа там, где можно помочь пользователю

Результат шага:

- приложение надёжнее не только технически, но и по пользовательскому поведению

### Шаг 6. Только потом добавлять Excel

Цель:

- не наращивать функциональность поверх ещё не доведённой архитектуры

Действия:

- сначала экспорт в Excel как более безопасный сценарий
- затем импорт с предпросмотром изменений
- не делать Excel основным форматом хранения

Результат шага:

- появляется обменный формат без разрушения текущей модели хранения
