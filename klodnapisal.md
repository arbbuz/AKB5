## Шаг 2 — тестовый проект (можно начать немедленно, без декомпозиции)

Создать `AKB5.Tests` под `net8.0` (не `net8.0-windows` — тесты должны бежать на Linux/CI). Три сервиса уже чистые и тестируемы прямо сейчас:

### `KnowledgeBaseService`
- `CanAttachSubtree` при разных глубинах поддерева
- `ReindexSubtree` корректно пересчитывает `LevelIndex` рекурсивно
- `DeleteNode` для корневых и вложенных узлов
- `CloneNode` не разделяет ссылки

### `JsonStorageService`
- round-trip: сохранить → прочитать → совпадает ли содержимое
- fallback на `.bak` при повреждённом основном файле
- `ValidateData` отклоняет конкретные некорректные структуры
- сохранение использует temp file (не пишет напрямую)

### `UndoRedoService`
- предел истории: при N+1 снапшоте удаляется старый
- redo-стек очищается при `SaveState`
- undo/redo при пустой истории возвращает null

## Шаг 3 — декомпозиция MainForm

**Ключевая проблема**: двойная синхронизация. `_workshopsData` не всегда актуальна — она обновляется в `SaveCurrentWorkshopState()` только перед сохранением/снапшотом. Это надо решить при декомпозиции.

**Что делать**: создать `AppController`, который:
- владеет `_config`, `_workshopsData`, `_currentWorkshop`, `_history`, `_clipboardNode`
- предоставляет методы `AddNode`, `DeleteNode`, `RenameNode`, `CopyNode`, `PasteNode`, `MoveNode` — только модель, ничего не знает о `TreeView`
- предоставляет методы `LoadFromFile`, `SaveToFile` — делегирует в `JsonStorageService`
- возвращает результат операции (успех/ошибка/изменённое состояние)

`MainForm` после этого становится тонким: вызывает контроллер → получает результат → обновляет `tvTree` и `lblInfo`.

**Порядок декомпозиции**:
1. Вынести операции дерева (`Add`, `Delete`, `Rename`, `Copy`, `Paste`, `Move`) — наиболее сложные, меняются чаще всего
2. State management (dirty-check, undo/redo coordination)
3. Файловые сценарии — последними, они в `MainForm` уже достаточно аккуратны
